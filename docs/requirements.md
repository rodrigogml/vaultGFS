# vaultGFS — Requirements

## Purpose

vaultGFS is a generic and extensible backup utility for multiple independent backup jobs. It supports GFS-style filesystem backups, MySQL logical dumps, manifests, SQLite cataloging and cron-line scheduling.

The project must be autonomous and not tied to any specific server.

## Core decisions

- Python project.
- Development path: `~/projects/vaultGFS`.
- Production path: `/opt/vaultGFS`.
- CLI names use normalized lowercase: `vaultgfs-*`.
- Linux runtime user/group: `vaultgfs`.
- SQLite catalog: `/var/lib/vaultgfs/catalog.db`.
- Sensitive config: `/opt/vaultGFS/config.toml`, not versioned.
- Config model: `config.toml.model`, versioned.
- No permanent daemon.
- Scheduling through cron-line schedules generated by `vaultgfs-reload`.

## Job model

The config must support multiple `[[jobs]]` blocks.

Standard attributes:

- `name`: unique job name.
- `enabled`: `true|false`. Default is `true` if absent, but examples must write it explicitly.
- `type`: initial values: `filesystem-gfs`, `mysql-dump`.
- `destination`: job destination.
- `skip_if_unchanged`: if `true`, do not create a new backup when no changes are detected.
- `compression_level`: optional override per job.
- schedule fields in cron-like format.

Cron-like format:

```text
minute hour day-of-month month day-of-week
```

Examples:

```text
0 1 * * * every day at 01:00
30 2 * * 0 every Sunday at 02:30
0 3 1 * * first day of each month at 03:00
0 1,13 * * * every day at 01:00 and 13:00
```

## Filesystem GFS jobs

A `filesystem-gfs` job backs up one source directory.

Required behavior:

- monthly FULL backup;
- weekly differential backup based on latest FULL;
- daily incremental backup based on latest successful backup;
- one fileserver job per internal folder;
- manifest JSON per run;
- metadata in SQLite;
- optional skip when unchanged;
- archives split by compression class.

Compression classes:

- `storage`: media/already compressed files, stored in tar without compression.
- `compressible`: fallback for everything else, stored in tar.zst.

Coverage rule:

```text
Every selected file must be included in exactly one class.
storage = configured extensions.
compressible = everything else.
```

## MySQL phase 1

Initial scope:

- full logical dumps only;
- one or more configured schemas per job;
- timestamped `.sql.zst` files;
- no GFS for MySQL yet;
- no binlog/PITR automation yet;
- dedicated MySQL user `vaultGFS`;
- `vaultgfs-reload` validates access.

System schemas must not be selected:

```text
mysql
information_schema
performance_schema
sys
```

Future MySQL roadmap:

- binlog archiving;
- point-in-time recovery;

## Executables

### `vaultgfs-backup`

Runs backup jobs.

Examples:

```text
vaultgfs-backup --job fileserver-docSafe --level full
vaultgfs-backup --job mysql-biserp
```

### `vaultgfs-reload`

Validates config, permissions, MySQL access and cron-line schedules. It proposes numbered changes and accepts:

```text
all
none
1,3,5
```

Future interactive restore utility.

## Resource control and monitoring

vaultGFS must support resource controls globally and per job. Job values override defaults.

Preventive controls:

- `compression_threads`: number of zstd worker threads. Do not use `0` for large production jobs because `zstd -T0` uses all available CPU threads.
- `compression_level`: zstd compression level. Levels above 19 require `--ultra`.
- `nice`: CPU scheduling niceness. Higher values mean lower priority. Recommended production value: `15`.
- `ionice_class`: Linux I/O scheduling class. Recommended production value: `2` (`best-effort`) or `3` (`idle`) for very conservative backups.
- `ionice_level`: I/O priority inside classes 1 and 2. Recommended production value for class 2: `7`.

`nice` lowers CPU priority; it is not a fixed CPU percentage limit. `ionice` lowers disk I/O priority. Subprocesses created by the backup process normally inherit these settings, including `tar`, `mysqldump` and `zstd`. For MySQL, this controls the `mysqldump` client process, but not the already-running `mysqld` server process. A large local dump can still put pressure on the MySQL server, buffer pool, storage and OS page cache.

Monitoring modes:

- `resource_monitor = "off"`: no monitoring.
- `resource_monitor = "passive"`: collect and log metrics while the job runs, without intervention.
- `resource_monitor = "active"`: collect metrics and react when configured limits are exceeded.

Metrics to monitor:

- CPU percentage of the child process group (`mysqldump`, `zstd`, `tar`, etc.). In Linux tools, CPU is cumulative across cores: `100%` means one full logical CPU. On a 20-thread host, `300%` means about three logical CPUs, not 300% of the whole machine.
- Memory percentage of the child process group.
- 1-minute system load average.
- Swap usage percentage.
- Output file size and growth rate.
- Elapsed time.

Active monitor reactions:

1. If limits are exceeded briefly, log the event.
2. If limits remain exceeded, temporarily pause subprocess groups with `SIGSTOP`.
3. After `pause_seconds`, resume with `SIGCONT`.
4. Repeat up to `max_pause_cycles`.
5. If limits remain exceeded for `abort_if_sustained_seconds`, abort the job, terminate subprocesses, remove partial artifacts and mark the catalog run as failed.

Suggested MySQL settings for large schemas:

```toml
compression_level = 22
compression_threads = 2
resource_monitor = "passive"
nice = 15
ionice_class = 2
ionice_level = 7
monitor_interval_seconds = 10
max_cpu_percent = 300
max_memory_percent = 35
max_load_1m = 16
max_swap_percent = 80
pause_seconds = 15
max_pause_cycles = 20
abort_if_sustained_seconds = 300
```

Future study: bandwidth/stream throttling for MySQL dumps. Current `mysqldump` help exposes protocol compression options (`--compress`, `--compression-algorithms`, `--zstd-compression-level`) but no direct bandwidth/rate limit option. Client/server protocol compression is not a throttling mechanism and is usually irrelevant or counterproductive on localhost Unix socket. Evaluate an optional pipeline throttle such as `pv -L` or another rate limiter later.

## Concurrency control

vaultGFS must prevent uncontrolled concurrent backup execution. This is required because per-process limits such as `compression_threads`, `nice` and `ionice` are not sufficient if several jobs start at the same time.

Global defaults:

```toml
max_concurrent_backups = 1
lock_wait_seconds = 10
```

Behavior:

- Before running any job, `vaultgfs-backup` must acquire one global backup slot.
- The number of slots is controlled by `max_concurrent_backups`.
- When all slots are busy, the new process waits and logs a `WAIT concurrency` message every `lock_wait_seconds`.
- The lock must be process-safe and automatically released if the process exits.
- Production recommendation for small servers or servers with MySQL dumps: `max_concurrent_backups = 1`.
- Each job log must include `QUEUE_INFO`, `RUN_START` and `RUN_END` with timestamps.
- `QUEUE_INFO` must report the queue entry time, effective start time and queue wait duration.
- `RUN_END.duration_seconds` must measure effective backup execution time only, excluding queue wait time.
